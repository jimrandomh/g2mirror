# g2mirror protocol

This document is for implementers of device drivers (e.g. the Even Realities
G2 smart-glasses driver) that display g2mirror terminal sessions. Protocol
version: **1**.

## Architecture

```
┌──────────────┐  websocket   ┌─────────────────┐  unix socket   ┌───────────────┐
│ device driver│◄────────────►│ g2mirror-server │◄──────────────►│ g2mirror      │
│ (you)        │   (JSON)     │ (gateway)       │  (NDJSON)      │ (one per      │
└──────────────┘              └─────────────────┘                │  wrapped app) │
                                                                 └───────────────┘
```

- Each `g2mirror <command>` invocation wraps one CLI app in a pty and exposes
  a **session socket** at `~/.g2mirror/<pid>-<sanitized-cwd>` (newline-
  delimited JSON, one client at a time).
- `g2mirror-server` is a websocket gateway: it authenticates the device,
  lists live sessions, and relays messages between the device and one
  session at a time per websocket connection.
- A driver normally talks only to the server. The session protocol is
  documented too because most messages are relayed through verbatim.

Transport security is deliberately out of scope: the server listens on the
address in `~/.g2mirror/config.json` (loopback by default) and is intended to
be reached over tailscale or an ssh tunnel. Run `g2mirror-server
--init-config` once to generate the config and the auth token (the token is
printed once; only its SHA-256 hash is stored).

## Framing

- **Websocket (device ↔ server):** each message is one JSON object in one
  websocket **text** frame.
- **Session socket (server ↔ wrapper):** each message is one JSON object on
  one `\n`-terminated line.

Every message has a string `type` field. Unknown fields must be ignored.
On either link, a message whose `type` the receiver doesn't recognize is an
error only where this document says so; drivers should tolerate and ignore
unknown message types for forward compatibility.

## Device ↔ server

### Handshake

The device's **first** message must be `init`:

```json
{"type": "init", "version": 1, "auth_token": "<token>",
 "device": "Jim's G2 glasses", "width": 96, "height": 24}
```

- `version` — protocol version; the server rejects mismatches.
- `auth_token` — the token generated by `--init-config`, in the clear
  (the transport is assumed private).
- `device` — free-text description, shown to humans.
- `width`/`height` — the device's terminal size **in character cells**
  (columns/rows). This depends on the device font size. It is fixed for the
  lifetime of the websocket connection; to change it, disconnect and
  reconnect with new values.

Success reply:

```json
{"type": "init", "version": 1}
```

Failure reply (`error`), then the server closes the connection:

```json
{"type": "error", "message": "authentication failed"}
```

### Server-scoped requests

**`{"type": "list"}`** — enumerate live sessions:

```json
{"type": "sessions", "sessions": [
  {"socket": "84210-_Users_jim_repos_myproj", "pid": 84210,
   "cwd_hint": "_Users_jim_repos_myproj"}
]}
```

`socket` is an opaque handle to pass to `connect`. `cwd_hint` is the
sanitized working directory — display it to help the user pick a session.

**`{"type": "connect", "socket": "84210-_Users_jim_repos_myproj"}`** —
attach to a session. The server dials the session socket and sends it a
session-protocol `init` derived from your websocket `init` (same `device`,
`width`, `height`). On failure you get an `error` message. On success the
next message you receive is the session's `connect` (below).

**`{"type": "disconnect"}`** — detach from the session. Reply:
`{"type": "disconnected", "reason": "requested"}`. You also receive a
`disconnected` message if the session ends or errors out
(`"reason": "session closed"`).

### Relay

While attached to a session:

- Every message the session sends is forwarded to you verbatim.
- Every message you send whose `type` is not `init`, `list`, `connect`, or
  `disconnect` is forwarded to the session verbatim (`view`, `unview`).

Sending a non-server-scoped message while not attached yields an `error`.

## Session protocol (wrapper ↔ its client)

Once attached, these are the messages you exchange (through the relay).

### From the session

**`connect`** — sent as soon as the wrapper accepts the connection:

```json
{"type": "connect", "version": 1, "pid": 84210, "command": "vim notes.md",
 "cwd": "/Users/jim/repos/myproj", "host_width": 143, "host_height": 40}
```

`host_width`/`host_height` are the size of the terminal the wrapper is
running in (cells), for display purposes.

**`snapshot`** — the immediate answer to `view`:

```json
{"type": "snapshot", "data": "<base64>"}
```

`data` is base64 of raw terminal bytes (VT100/xterm escape sequences).
Feed the decoded bytes to a terminal emulator of exactly the `width` ×
`height` you declared in init, **reset/cleared first** — the snapshot begins
with a clear and repaints everything, including cursor position/visibility
and input modes. Its content is the bottom-left crop of the app's screen at
the moment you asked to view (the app itself repaints at your dimensions a
moment later; that arrives as `output`).

**`output`** — incremental terminal bytes while viewing, same base64
encoding. Apply them to the same emulator, in order, after the snapshot.
The stream is a diff against the emulator's current state, so messages must
not be dropped or reordered; if you lose state, send `view` again for a
fresh snapshot.

**`exit`** — the wrapped app exited; the wrapper closes the socket (you'll
then get `disconnected` from the server):

```json
{"type": "exit", "status": 0}
```

`status` is the exit code, or `null` if the app died to a signal.

**`error`** — protocol violation report; the wrapper closes the connection
after sending it.

### To the session

**`init`** — first message; the server sends this for you when relaying.

**`{"type": "view"}`** — start viewing. Effects, in order:

1. The wrapped app's pty is resized to your `width`×`height` (the app gets
   SIGWINCH and will typically repaint itself).
2. You immediately receive `snapshot`.
3. Terminal output streams to you as `output` messages (usually starting
   with the app's SIGWINCH repaint) until `unview` or disconnect.

Sending `view` while already viewing is allowed and simply re-snapshots
(use it to resynchronize).

**`{"type": "unview"}`** — stop viewing: output streaming stops and the app
is resized back to the host terminal. Idempotent. Use this whenever the
glasses UI navigates away from the terminal so the app returns to full size
for the person at the keyboard.

## Terminal data notes

- The byte streams use standard VT100/xterm escape sequences (CSI cursor
  addressing, SGR colors including 256-color and truecolor, alternate
  screen, bracketed paste / mouse / application-key mode toggles, and
  synchronized-update markers `CSI ? 2026 h/l` you may use for flicker-free
  rendering, or ignore).
- Any headless terminal-emulator library that maintains a screen grid works
  (the reference implementation uses Rust's `vt100` crate; xterm.js headless
  works too).
- Keyboard input from the device is **not supported** in protocol version 1
  — there is no message for it, and input modes are informational only.
- One client per session: a second connection to a busy session socket gets
  `error` and is closed.

## Liveness and cleanup

- The wrapper removes its socket file on exit. `g2mirror-server` also
  removes, at startup, socket files that are both older than 1 hour and
  owned by a PID that no longer exists.
- `list` only reports sockets whose owning PID is alive; a `connect` can
  still fail if the session exits in between.
- The wrapper drops clients that stop reading for ~5 seconds; keep
  consuming messages promptly while viewing.

## Typical driver flow

```
connect ws://host:8737
→ {"type":"init","version":1,"auth_token":"…","device":"Jim's G2","width":96,"height":24}
← {"type":"init","version":1}
→ {"type":"list"}
← {"type":"sessions","sessions":[…]}          # user picks one
→ {"type":"connect","socket":"84210-_Users_jim_repos_myproj"}
← {"type":"connect","version":1,"pid":84210,…} # attached
→ {"type":"view"}                              # glasses look at terminal
← {"type":"snapshot","data":"…"}               # paint immediately
← {"type":"output","data":"…"}                 # keep applying
…
→ {"type":"unview"}                            # glasses look away
…
→ {"type":"view"}                              # glasses look back
← {"type":"snapshot","data":"…"}
…
← {"type":"exit","status":0}
← {"type":"disconnected","reason":"session closed"}
```
